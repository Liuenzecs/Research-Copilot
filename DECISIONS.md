# 关键决策记录

更新时间：2026-03-29

本文件用于记录当前主线下的重要产品、架构和协作决策。默认追加新记录，不随意覆盖旧结论；如果决策被推翻，应明确写出变更原因。

## 2026-03-29：加入项目后默认自动触发既有 `fetch_pdfs` 任务，而不新起下载系统

状态：已生效

背景：

- 用户的核心阻塞不是“没有下载按钮”，而是“论文刚加入项目后还不能立刻进入可读状态”，导致需要再手动补一次 PDF 下载。
- 当前后端已经有成熟的 `fetch_pdfs` 项目动作、任务流和 PDF 状态字段，如果为自动下载再新起一套后台队列，会徒增维护分叉。
- 阅读器此前只知道“有没有下载成功”，不知道“现在是在排队、下载中还是失败”，因此空态提示过于笼统。

决策：

- 保留现有 `fetch_pdfs` 作为唯一 PDF 补齐链路；项目工作台和项目搜索工作台在“加入项目”成功后，默认自动触发这一动作。
- `fetch_pdfs` 任务启动前先把相关论文标记为 `queued`，任务执行时切到 `downloading`，完成后再回到既有 `downloaded / error / landing_page_only / missing` 状态集合。
- 阅读器响应补充 `pdf_status` 与 `pdf_status_message`，无 PDF 时按当前状态给出更贴近现场的提示；当状态为 `queued` 或 `downloading` 时，前端自动轮询刷新。
- 第一轮不引入新的独立下载服务、数据库表或复杂并发控制，只复用现有项目动作与任务回流机制。

影响：

- “加入项目”到“进入可读状态”之间的断裂明显缩短，大多数论文会在用户继续筛选或回到工作台的过程中自动补齐 PDF。
- 阅读器空态从“统一报没下载”变成“解释当前卡在哪一步”，用户更容易判断是稍等、刷新还是手动重试。
- 这条链路仍然集中在现有项目任务系统里，后续如果要补并发控制、失败重试或文库级下载视图，可以沿同一条主线演进。

## 2026-03-29：`/projects` 默认进入最近打开项目，项目管理改为显式管理视图

状态：已生效

背景：

- 当前桌面端的主要使用方式已经收敛到“围绕一个持续推进中的项目做检索、阅读和写作”，每次启动先落到项目管理页会打断主流程。
- 用户明确希望“项目”继续表示“回到我正在推进的项目”，而不是“回到可创建新项目的总入口”，但同时仍需要保留项目管理与新建能力。
- 现有后端已经维护 `last_opened_at`，可以直接表达“最近打开项目”，不需要额外引入新的默认项目配置字段。

决策：

- `/projects` 不再默认停在项目管理页，而是优先进入最近打开项目的工作台；只有在没有任何项目时，才展示项目首页。
- 项目管理入口保留为显式视图 `/projects?view=manage`，用查询参数区分管理态，从而保持既有路由语义和主路径稳定。
- 顶部品牌入口、顶部“项目”导航与侧边栏“项目”入口统一优先指向最近打开项目；项目工作台右上角补一个弱化的“项目列表”按钮，用于进入管理视图。
- 第一轮不新增“启动时进入上次项目”设置项，先把默认行为直接收口到更贴近真实使用习惯的路径；如后续出现明确关闭需求，再补设置项。

影响：

- 应用启动、跨模块返回和点击“项目”时都会更接近“回到当前工作现场”，减少先落到管理页再二次进入工作台的打断。
- 项目管理从主入口退居为显式入口，但仍然可达，且不需要新增新路由或破坏 `/projects` 作为项目主入口的既有语义。
- 这一决策复用了现有后端最近打开时间排序，避免为了“默认项目”概念引入新的持久化配置和同步复杂度。

## 2026-03-29：主舞台范围说明要直接复用接续聚焦提示，而不只保留顶部说明

状态：已生效

背景：

- 主舞台已经补上了范围来源、范围规模和排序语义，但“当前接续聚焦到底筛掉了什么”此前仍主要留在论文池顶部的另一句提示里。
- 这会让主舞台虽然已经更完整，却还是缺最后一小块解释，用户想彻底看懂当前范围时仍要回顶部做最后一次对照。

决策：

- 主舞台范围说明直接复用 `activeReaderFocusMeta.hint`，把当前接续聚焦的解释性文案同步到同一区域。
- 聚焦提示继续跟随当前接续状态动态变化，与来源、规模和排序语义保持同一套前端派生逻辑。
- 这一步继续只消费现有前端文案和派生状态，不新增新的后端字段或说明模型。

影响：

- 用户在主舞台就能一次性看懂“范围来自哪里、范围有多大、怎么排序、聚焦筛掉了什么”。
- 主舞台的阅读解释块更接近完整闭环，后续再做视觉收口时不需要再补新的语义缺口。

## 2026-03-29：主舞台范围说明要直接复用排序语义，而不只展示结果列表

状态：已生效

背景：

- 主舞台已经能说明当前范围来自哪里、范围里有多少篇，但如果用户想知道“这一组为什么这样排”，仍然需要回到论文池顶部看另一句排序说明。
- 这会让主舞台继续停留在“只给结果，不解释顺序”的状态，阅读判断还是会被迫在两个区域之间来回切换。

决策：

- 主舞台范围说明直接复用论文池顶部已有的 `projectReaderOrderHint(...)` 排序提示。
- 排序提示跟随当前接续聚焦同步变化，在默认范围、优先回看、继续阅读和先留在池里等状态下保持同一套语义。
- 这一步继续只消费前端已有派生状态，不新增新的后端排序字段或说明模型。

影响：

- 用户在主舞台就能同时看懂“当前范围为什么这样”和“这一组为什么这样排”，不必再来回对照顶部说明。
- 主舞台开始承担更完整的阅读解释职责，范围来源、范围规模和排序语义终于能放在一起读完。

## 2026-03-29：主舞台范围摘要要直接补足总量，而不只给局部候选数

状态：已生效

背景：

- 主舞台已经能告诉用户当前范围来自哪里，也能就地放宽范围，但摘要条此前仍然主要显示“已保存会话多少篇、优先回看多少篇”。
- 当用户想快速判断当前范围到底大不大、还有多少篇根本没开始读时，仍然需要自己拿候选数和下方列表心算，扫读负担偏高。

决策：

- 主舞台摘要条直接补上当前范围总篇数、已保存会话数、待安排篇数和优先回看篇数。
- 这组总量跟随当前范围同步刷新，在默认范围、回跳范围和 0 结果范围下都保持同一套表达。
- 继续复用前端现有派生状态，不为这一步引入新的后端统计字段。

影响：

- 用户在主舞台就能直接判断当前范围大小和阅读准备度，不需要再靠候选卡片数量反推。
- 主舞台摘要开始承担“概览当前范围规模”的职责，和范围来源、范围收口形成更完整的一组阅读上下文。

## 2026-03-28：主舞台里的空范围要被解释成“范围过窄”，而不只是一个普通空态

状态：已生效

背景：

- 主舞台已经能显示范围来源和范围收口按钮，但当“智能视图 + 阅读接续聚焦”组合把当前范围缩到 0 篇时，原来的文案仍然只会说“当前范围还没有论文可安排阅读接续”。
- 这类空态对用户来说不够准确，因为问题往往不是没有阅读接续，而是当前范围被压得太窄，下一步应该放宽范围而不是怀疑数据丢失。

决策：

- 主舞台在非默认范围且当前范围为 0 篇时，优先显示“范围过窄”的解释型提示，而不是继续复用普通空态文案。
- 提示文案根据当前是智能视图过窄还是“智能视图 + 接续聚焦”叠加过细，给出更贴近当前状态的放宽建议。
- 这项提示继续依附在主舞台现有范围说明和范围按钮之下，不新增新的空态面板或另一套恢复入口。

影响：

- 用户看到 0 篇时更容易理解这是筛选过窄，而不是阅读接续链路又断掉了。
- 主舞台的空态开始承担“解释为什么空”和“告诉用户下一步该怎么放宽”的职责，阅读流程更连续。

## 2026-03-28：主舞台里的范围收口操作要就地可用，而不是只留在论文池顶部

状态：已生效

背景：

- 主舞台现在已经能解释当前范围来自哪里，但如果用户想立刻放宽范围，仍然需要再回到论文池顶部的来源提示条去找按钮。
- 这会让“看懂当前范围”和“调整当前范围”被迫拆在两个区域里，继续增加滚动和视线来回。

决策：

- 主舞台范围说明下方直接镜像论文池顶部现有的范围收口操作。
- 这些操作继续复用同一套 `清除接续聚焦 / 清除智能视图 / 回到默认范围` 逻辑，不新增新的范围状态分支。
- 主舞台承担“解释范围”和“就地收口范围”两件事，减少用户必须在两个区域之间切换。

影响：

- 用户在主舞台确认完当前范围后，可以直接在眼前完成放宽或复位，不需要再回顶部找按钮。
- 论文池顶部和主舞台继续共用同一套范围动作语义，后续维护时更不容易出现一边能撤销、一边不能撤销的分叉。

## 2026-03-28：主舞台里的阅读建议要复用论文池同一套范围来源语义

状态：已生效

背景：

- 论文池顶部已经能解释当前范围来自状态中心、阅读接续聚焦还是智能视图，但主舞台里的“阅读接续建议”此前只显示当前范围结果，没有同步告诉用户这组候选为什么会出现。
- 这会让主舞台摘要和论文池顶部提示重新分裂成两套上下文：列表知道自己为什么缩小了，主舞台却像在描述另一个范围。

决策：

- 主舞台“阅读接续建议”直接复用论文池现有的范围来源语义，不额外创建新的来源字段或另一套说明文案。
- 主舞台同时展示来源标签、当前范围组合和来源解释，让用户在同一块区域里就能看懂“现在为什么只剩这组论文”。
- 主舞台统计区补充“范围来源”迷你指标，进一步降低扫读时的上下文确认成本。

影响：

- 用户在主舞台就能理解当前候选论文的来历，不必再在顶部提示条和主舞台摘要之间来回对照。
- 后续如果继续调整范围来源或回跳链路，只需要维护同一套来源语义，主舞台和论文池更不容易再次说成两套话。

## 2026-03-28：搜索工作台要直接暴露阅读接续状态，而不是把阅读入口埋在详情与 AI 理由下面

状态：已生效

背景：

- 用户在项目搜索台里重新看到一篇论文时，真正要先判断的是“我是不是已经读过、现在该继续读还是优先回看”，而不是先重新打开详情区再慢慢确认。
- 现有阅读器已经有本地阅读会话和待回看段落，但这些线索此前没有回流到搜索结果本身，导致搜索与阅读之间仍然存在明显的上下文断裂。

决策：

- 搜索工作台直接消费前端本地的 `paperReaderSession`，不为这一轮新增后端阅读同步模型。
- 搜索结果卡片如果存在阅读会话，就直接展示上次停留页码、阅读模式、保存时间，以及待回看段落数。
- 右侧检查面板新增独立的“阅读接续”卡片，把阅读入口从 AI 理由区拆出来；入口文案根据状态在“打开阅读器”“继续阅读”“优先回看”之间动态切换。
- 已保存搜索相关测试标识与阅读入口测试标识保持分离，避免随着搜索台交互加深而再次出现选择器碰撞。

影响：

- 用户在搜索阶段就能直接判断这篇论文是否值得马上续读，减少“先搜到、再点详情、再回忆读到哪”的摩擦。
- 搜索台开始承担阅读接续分发的职责，而阅读器继续承担具体的段落回看、批注与证据提取职责，分层更清晰。

## 2026-03-28：项目论文卡片要直接表达阅读决策，而不只给一个统一的阅读器入口

状态：已生效

背景：

- 项目工作台的论文池此前已经能显示是否存在阅读会话，但如果入口始终只写成“继续阅读”或“打开高级阅读器”，用户仍然要自己再判断这篇论文是该马上回看，还是只是普通续读，还是其实可以先留着稍后再说。
- 当阅读接续线索已经从阅读器回流到项目工作台后，论文卡片本身如果不表达“现在该怎么处理”，阅读判断仍然会被迫绕到右侧状态中心或搜索台里完成。

决策：

- 每张项目论文卡片都直接给出一个阅读接续判断：`优先回看`、`继续阅读`、`先留在池里`。
- 项目论文卡片上的主阅读入口文案跟随判断动态切换，不再让有待回看段落的论文继续显示成笼统的“继续阅读”。
- 论文池主舞台单独增加当前智能视图下的“阅读接续建议”摘要，让用户不切出当前舞台也能快速看到当前视图里最该先处理的阅读对象。

影响：

- 用户在项目论文池里就能直接做阅读优先级判断，而不是必须跳去状态中心或再次打开搜索工作台。
- 论文卡片开始承担“告诉用户现在该怎么读”的职责，而右侧状态中心继续承担跨视图的全局阅读回流摘要职责。

## 2026-03-28：项目论文池要按阅读接续状态分层，而不是继续平铺所有论文

状态：已生效

背景：

- 当项目里已经积累了多篇论文，仅仅给每张卡片补上阅读状态还不够；如果列表仍然平铺，用户还是要自己在一长串卡片里逐张扫读，才能找到最该先回去的那几篇。
- 这会让前面主舞台摘要里给出的“优先回看候选”重新断掉，因为用户仍然要回到论文池列表里手动对照具体卡片。

决策：

- 论文池列表直接按 `优先回看`、`继续阅读`、`先留在池里` 三层分组显示。
- 分组排序优先复用现有本地阅读会话：待回看组按待回看段落数与最近保存时间排序，继续阅读组按最近阅读时间排序，未建立会话的论文继续保留项目原始排序。
- 这项分组只消费前端已有派生状态，不新增新的后端排序字段或额外查询。

影响：

- 用户进入论文池后，最该先回去看的论文会先被顶出来，减少在长列表里重新找优先级的摩擦。
- 主舞台的阅读接续建议和论文池列表本身终于使用同一套优先级语义，阅读判断更连贯。

## 2026-03-28：项目论文池里的阅读接续状态要能被直接聚焦，而不只停留在分组展示

状态：已生效

背景：

- 论文池已经按“优先回看 / 继续阅读 / 先留在池里”分组后，用户虽然更容易扫到重点，但一旦切换智能视图，仍然可能需要在当前范围里重新扫读整列卡片，才能只看某一种阅读接续状态。
- 如果主舞台、论文池列表和批量选中没有共享同一个“当前范围”定义，用户会看到摘要说的是一套范围，列表和批量动作又落在另一套范围上，继续产生上下文断裂。

决策：

- 在当前智能视图之上，增加一层前端本地的“阅读接续聚焦”筛选：`全部接续状态`、`只看优先回看`、`只看继续阅读`、`只看先留在池里`。
- 论文池顶部显式展示当前聚焦下的排序语义，让用户知道优先回看按待回看段落数与最近保存时间排序，继续阅读按最近阅读时间排序，先留在池里沿用项目原始排序。
- 主舞台摘要、论文池列表和批量选中统一采用“当前范围 = 智能视图 + 阅读接续聚焦”的语义，而不是各自维护不同的范围解释。

影响：

- 用户切换智能视图后，可以继续一键压缩到真正关心的阅读接续状态，不必每次重新扫读整个论文池。
- 阅读建议、列表分组和批量动作终于共享同一个范围模型，当前看到什么、准备处理什么会更一致。

## 2026-03-28：状态中心里的阅读回流要负责把用户带回论文池，而不只负责提醒

状态：已生效

背景：

- 右侧“阅读回流”卡片已经知道哪些论文值得优先回看，但如果它只能显示摘要和候选链接，用户看见提醒后仍然需要自己回到论文池再切一遍范围，才能处理同组论文。

## 2026-03-28：论文池要解释当前范围的来源，而不只显示结果本身

背景：

- 论文池顶部在非默认范围下显示来源提示条，明确说明当前范围来自 `智能视图`、`阅读接续聚焦` 还是 `状态中心`。

- 来源提示条内直接提供“一键回到默认范围”入口，恢复到 `全部论文 + 全部接续状态`。

影响：

- 用户从任意入口跳回论文池后，更容易理解自己当前看到的是哪一组论文，不必靠回忆倒推。

## 2026-03-28：论文池范围的退出动作要支持逐层撤销，而不只支持整页重置

背景：

- 当论文池已经叠加了“智能视图 + 阅读接续聚焦”两层范围时，如果唯一退出动作只有“回到默认范围”，用户就会被迫把两层一起丢掉，即便他其实只想退掉其中一层。
- 这会制造新的跳变感：刚刚还在一个接近目标的范围里，只是想看得稍微宽一点，结果整页直接回到了起点。

决策：

- 在范围提示条里同时提供三类动作：`只清除阅读接续聚焦`、`回到全部论文但保留当前接续聚焦`、`回到默认范围`。
- 这三类动作分别对应撤销一层聚焦、撤销一层智能视图、撤销全部范围，而不是继续把“退出当前范围”简化成一个粗糙的总重置按钮。
- 为智能视图切换补充稳定测试标识，确保“叠加范围后逐层撤销”的回归可以被持续验证。

影响：

- 用户可以更平滑地从窄范围退回到宽范围，而不是每次都被迫从头重新筛一次。
- 论文池范围控制从单一重置升级为分层撤销，阅读探索会更像连续缩放，而不是频繁跳回起点。

## 2026-03-28：项目工作台要主动给出“优先回看候选”，而不只停留在笼统的继续阅读入口

影响：

- 用户从阅读器回到项目工作台后，可以更快判断“应该先回哪篇继续读”，减少重新翻卡片和重新定位的摩擦。
- 工作台开始承担跨论文阅读接续的角色，而单篇阅读器继续承担具体的段落回看与批注处理职责，分层更清晰。

- 搜索命中列表继续保留，但默认只展示前 8 条；更长的命中序列优先通过顺序跳转完成，而不是把导航卡无限拉长。

影响：

- 长文档搜索从“找一次”变成了“沿着命中顺序往前扫读”，更贴近真实阅读习惯。
- 定位栏本身开始承担状态反馈职责，用户不需要频繁在正文、导航卡和页码之间来回确认当前位置。

## 2026-03-28：搜索命中要直接落到正文段内，而不只停留在导航反馈

状态：已生效

背景：

- 当用户已经通过定位跳到目标段落后，如果正文里没有直接标出命中的关键词，仍然要重新扫读整段内容，阅读中断并没有真正消失。
- 搜索命中、当前焦点和批注状态已经同时存在于阅读器里，如果继续把搜索反馈只放在定位栏和徽标上，用户很难快速分辨“我为什么来到这里”。

决策：

- 搜索命中的关键词直接在正文、标题、图注和公式段落内做局部高亮，而不是只在外层卡片上标记“搜索命中”。
- 高亮样式继续收敛在 `paper-reader-enhancements.css`，保持阅读器实验性视觉增强局部化，不扩散回全局样式。
- 当前焦点段落与搜索命中的视觉层次要能同时成立，高亮用于“看到词”，焦点态用于“看到当前位置”，两者不互相替代。

影响：

- 用户跳到命中段落后可以立刻看到关键词落点，减少“已经跳到了，但还要重新找一遍”的摩擦。
- 后续继续加强批注段落可视化时，可以直接在正文层叠加多种阅读反馈，而不必再把所有状态都塞回侧栏或摘要提示里。

## 2026-03-28：批注反馈要在正文段内表达“记过什么”，而不只标记“记过”

- 如果段落上只有一个泛化的“已批注”状态，用户虽然知道这里动过笔记，但仍然不知道到底记了什么，回看时还是得重新打开右侧工作台或点回批注列表。
- 阅读器已经支持搜索命中高亮和当前焦点强化；批注反馈如果继续停留在抽象标签层，会成为最弱的一层状态，无法真正承担“帮助复读和复核”的作用。

决策：

- 已批注段落在正文流里除了保留状态徽标，还要直接显示批注条数，避免多次记录被压扁成同一种“已批注”。
- 段落内新增轻量的“最近批注”摘录卡片，把最近一条批注内容压缩成可扫读反馈，帮助用户在正文里快速想起这段记过什么。
- 这类增强继续留在 `paper-reader-enhancements.css` 和阅读器段落渲染层内实现，不把批注可视化逻辑扩散回全局布局。

影响：

- 用户通过键盘、搜索或批注回跳回到某段时，可以直接在正文里看到这段的批注密度和最近一条记录，降低重复打开工作台的成本。
- 后续如果继续把批注上下文带入焦点摘要，也可以基于同一套段落级批注摘要继续演进，而不必重新设计另一套数据表达。

## 2026-03-28：顶部焦点摘要只补充当前段落的批注上下文，不复制整块批注工作台

状态：已生效

背景：

- 在正文里补上“最近批注”之后，键盘跳段和搜索定位虽然已经更容易看见批注，但用户视线仍然经常先落在顶部焦点摘要区；如果这里完全没有批注上下文，还是会出现“已经跳到了，但还得回正文里再找一遍”的摩擦。
- 另一方面，顶部摘要如果直接搬运完整批注列表，又会和右侧批注工作台形成重复，削弱当前阶段想保持的轻量阅读节奏。

决策：

- 顶部焦点摘要只在“当前焦点段落确实存在批注”时补充一张轻量上下文卡片，展示最近一条批注摘录和批注条数。
- 这张卡片必须跟随当前焦点段落同步切换；离开该段后，旧批注上下文不能残留在顶部摘要里。
- 焦点摘要只负责回答“当前这段记过什么”，完整批注整理、回跳和沉淀仍然留在右侧批注工作台里。

影响：

- 用户通过键盘连续阅读时，顶部摘要就能直接给出当前段落的批注上下文，减少在摘要区和正文区之间来回找线索的成本。
- 顶部摘要和正文段内摘录形成“先看概览，再看正文细节”的分层，而不是两块区域互相复制同一批信息。

## 2026-03-28：批注回跳时，顶部摘要优先说明“由哪条批注带回”，而不是继续显示泛化批注概览

背景：

- 当用户从批注工作台或最近批注快捷入口回到正文时，单靠“已从批注回跳到对应段落”这样的最近动作文案还不够，用户仍然要自己判断究竟是被哪条批注带回来的。
- 如果顶部摘要在这种情况下继续展示泛化的“当前段落批注”卡片，就会和正文段内摘录形成重复，却没有回答“这次为什么会回到这里”。

决策：

- 批注回跳会把来源标签和对应批注内容一起带进顶部摘要，例如“待处理批注”“已沉淀批注”或“最近批注快捷入口”。
- 当存在明确的批注回跳来源时，顶部摘要优先显示“当前由批注带回”卡片，不再同时展示泛化的“当前段落批注”概览。
- 泛化批注概览继续保留给普通段落浏览场景；只有在用户刚刚沿着批注线索回到正文时，才切换成来源型摘要。

影响：

- 用户从批注线索回到正文后，可以更快确认“我是被哪条记录带回来的”，减少在摘要区、正文和工作台之间来回对照。
- 顶部摘要开始承担“说明当前回跳来源”的职责，而正文段内摘录继续承担“展示这段最近记过什么”的职责，分工更清晰。

## 2026-03-27：`paragraph_id` 深链只消费一次，不能反向覆盖阅读器内部导航反馈

状态：已生效

背景：

- 阅读器内部会在跳段后把 `paragraph_id` 同步进 URL，保证刷新后还能回到当前段落。
- 如果把这次内部 URL 同步再误判成新的“外部指定段落”入口，焦点摘要和最近动作就会被重放成旧文案，用户会看到“已经通过章节导航跳了过去，但提示又变回指定段落”的迟滞感。

决策：

- `paragraph_id` 的“指定段落”语义只在进入当前论文时消费一次。
- 阅读器内部导航仍然可以继续更新 URL，但不再重放“已定位到指定段落”的初始反馈。
- 最近动作提示由真实触发导航的动作负责写入，例如关键词定位、章节导航、图像锚点、批注回跳和页码切换。

影响：

- URL 仍然保留了刷新与分享定位的价值，但不会再污染阅读器内部的动作反馈。
- 焦点摘要能更准确地反映用户刚刚执行的动作，减少长文档里“位置对了、文案没跟上”的割裂感。

## 2026-03-27：阅读会话按论文粒度保存在前端本地

- 如果未来需要跨设备同步阅读进度，再评估是否引入服务端会话模型。

- 关闭浮层后的键盘连续性明显更强，用户不用再多做一次回收焦点的动作。
- 这让桌面阅读器的层级心智更清晰：打开一层，就操作这一层；关闭一层，就回到下一层。

## 2026-03-27：`Esc` 在阅读器输入控件里应是非破坏性的退出动作

状态：已生效

背景：

- 阅读器里已经有定位输入框、批注输入框和多种下拉控件，但此前按 `Esc` 时，它们主要只是“吞掉快捷键”，并不会把键盘控制权交还给阅读器。
- 用户从输入态切回阅读态时，常常还要再点一下空白处，这在 Windows 桌面环境里很不顺。

决策：

- 在阅读器里，定位输入框、批注输入框和下拉控件获得焦点时，按 `Esc` 应优先退出当前输入态并把焦点交还给阅读壳层。
- 这个退出动作是非破坏性的：批注草稿不丢失，下拉值不被重置。
- 定位输入框额外清理临时错误提示，避免用户离开输入态后还保留过时的红色反馈。

影响：

- 用户可以更自然地从“正在输入”切回“继续阅读”，而不需要重新找点击目标。
- 快捷键链路会更完整：`/` 进入定位，`Esc` 退出定位，随后立即继续翻页、切模式或跳段。
- 这让阅读器的输入控件更接近桌面工具里的“临时接管键盘”语义，而不是一旦进入就很难退出。
