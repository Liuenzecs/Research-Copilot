# 关键决策记录

更新时间：2026-03-27

本文件用于记录当前主线下的重要产品、架构和协作决策。默认追加新记录，不随意覆盖旧结论；如果决策被推翻，应明确写出变更原因。

## 2026-03-27：阅读会话按论文粒度保存在前端本地

状态：已生效

背景：

- 阅读器已经承担深读、翻译、批注和证据提取，但刷新或重新进入后缺少“继续上次阅读”的能力。
- 这轮优化希望先尽快改善单机桌面体验，而不是先引入新的后端状态模型。

决策：

- 阅读器按 `paperId` 在前端本地保存阅读会话。
- 当前会话内容至少包括：视图模式、页码、段落和缩放比例。
- 本地阅读会话优先使用 `localStorage` 实现，不为此单独增加数据库或 API 结构变更。
- 如果路由中显式带有 `paragraph_id` 等深链定位参数，则深链优先于本地恢复结果。

影响：

- 用户刷新页面或重新进入同一篇论文时，可以直接回到上次阅读位置。
- 本轮实现保持了低风险、低侵入，不影响现有后端 schema 和桌面运行时数据规则。
- 如果未来需要跨设备同步阅读进度，再评估是否引入服务端会话模型。

## 2026-03-27：结构化导航先做轻量快捷跳转，不先上重型侧栏

状态：已生效

背景：

- 阅读器已经需要支持章节、图像、批注、搜索命中的快速跳转，但当前更重要的是尽快降低查找成本。
- 现阶段如果直接引入复杂的树状目录、多栏侧边导航或更重的阅读器布局，会增加实现成本和界面负担。

决策：

- 先在阅读器主区内提供轻量化的结构化导航卡。
- 第一版优先覆盖：章节导航、图像跳转、批注回跳、搜索命中快速跳转。
- 暂不为了这项能力引入新的重型 outline 树、固定多栏信息架构或更复杂的阅读器壳层。

影响：

- 用户能更快在长论文中移动，而不会明显增加界面复杂度。
- 后续如果导航项继续增长，再评估是否升级成更完整的目录侧栏方案。

## 2026-03-27：选区要在翻译、批注、证据之间持续保留

状态：已生效

背景：

- 阅读器里的高频动作并不是孤立发生的，用户常常会先选一段，再翻译、写批注、最后决定是否加入证据。
- 如果每做一步都要重新选区，会明显破坏阅读连续性。

决策：

- 当前选区在进入翻译或批注流后应尽量保留为可继续使用的引用上下文。
- 选区浮动工具条优先提供连续动作，而不是把翻译和证据操作拆成分散入口。
- 批注面板要能承接前一步留下的引用原文，而不是要求用户再次选中。

影响：

- 选区从一次性鼠标动作变成了阅读链路中的短时上下文。
- 这项设计会降低重复选择文本的成本，也让翻译结果更容易自然沉淀成批注或证据。

## 2026-03-27：待回看段落先作为本地阅读状态实现

状态：已生效

背景：

- 用户在阅读中常会遇到“现在先记住，稍后回来看”的段落。
- 这类状态对阅读连续性很重要，但暂时不一定需要先上后端模型。

决策：

- 待回看段落先作为阅读器本地状态实现，并随阅读会话一起持久化。
- 待回看优先服务当前单机阅读流，不先引入数据库字段或项目级 API。
- 如果后续项目工作台需要看到这些状态，再评估把它们同步回更上层的数据模型。

影响：

- 用户现在可以在阅读器里留下轻量的“回看锚点”，并在下次进入时继续接上。
- 这让阅读进度表达更完整，也为后续项目级阅读状态同步留出了演进空间。

## 2026-03-27：项目工作台先消费前端本地阅读会话，不为 1E 引入后端同步

状态：已生效

背景：

- 阅读器已经有按论文保存的本地阅读会话，但项目工作台里的论文池仍然主要展示后端返回的静态项目数据。
- 当前阶段的核心问题是“让用户回到项目工作台时还能看见阅读上下文”，而不是马上建设一套新的服务端阅读状态模型。

决策：

- 阶段 1E 先由项目工作台直接读取前端本地的 `paperReaderSession`。
- 论文卡片优先回流的状态包括：是否存在阅读会话、最近保存时间、待回看段落数、最近恢复入口。
- 对已有本地阅读会话的论文，项目工作台优先提供“继续阅读”入口，并回到记住的段落。
- 本阶段不新增数据库字段、后端聚合逻辑或跨设备同步语义。

影响：

- 阅读器和项目工作台之间的状态联动可以先快速落地，且不影响现有 API 与 schema。
- 这让“从项目来、回项目去”的路径更连续，也给后续是否升级为项目级持久化状态留出了观察窗口。
- 如果未来需要跨设备同步或团队共享阅读状态，再评估把这些状态提升到后端模型。

## 2026-03-27：辅助文本排版优先通过收窄阅读列和页内概览优化，不先引入主题系统

状态：已生效

背景：

- 辅助文本模式已经承载翻译、批注和证据提取，但用户阅读长段正文时仍容易感到视线发散、页内重新定向成本高。
- 当前阶段更需要先把默认阅读体验做顺，而不是提前扩展字体、密度、主题等一整套偏好系统。

决策：

- 优先在默认辅助文本模式中收窄正文阅读列，拉开标题、正文、图注、公式之间的视觉节奏。
- 在页内增加轻量级“阅读概览”，帮助用户快速判断这一页主要是正文、结构锚点还是图示 / 公式密集页。
- 页内概览优先给出阅读提示和状态摘要，而不是新增复杂的配置面板或个性化主题系统。
- 本阶段样式增强继续集中在 `paper-reader-enhancements.css`，避免把阅读器实验性优化扩散回全局样式。

影响：

- 默认阅读体验会先变得更稳、更可连续使用，且改动范围保持在阅读器局部。
- 这为后续是否需要字体大小、密度、主题偏好等个性化能力留出了观察窗口。
- 如果后续用户仍明确需要更强的个性化控制，再评估把默认优化升级成可配置主题系统。

## 2026-03-27：批注工作台的“待处理 / 已沉淀”先用待回看与项目证据联动定义

状态：已生效

背景：

- 阅读器里已经有批注、待回看和加入项目证据三类相关动作，但之前只是在当前页平铺批注，用户很难快速判断下一步该处理什么。
- 这一阶段的目标是先让批注整理成本降下来，而不是马上引入新的“批注任务”后端模型。

决策：

- 阅读器侧批注工作台先按三块组织：`待处理批注`、`当前页批注`、`最近已沉淀`。
- 在项目上下文里，凡是还没有进入当前项目证据的批注，优先视为“待处理”。
- 无论是否在项目上下文里，被标记为“待回看”的批注都应继续留在待处理集合中，作为更强的跟进信号。
- 已沉淀状态先通过“已进入当前项目证据”或“未被标记待回看”来推断，不为 1G 单独增加数据库字段或额外 API。
- 批注保存后直接回填前端当前 reader/query 缓存，保证工作台状态即时更新，再用后台失效刷新兜底同步。

影响：

- 用户能更快看出哪些批注还要补证据、补心得或回看原文，哪些已经完成了阶段性沉淀。
- 这一方案延续了前面几轮“先做前端低风险联动，再观察是否需要上后端模型”的策略。
- 如果后续需要跨设备同步批注处理状态，或需要更细的任务流转，再评估是否把“待处理 / 已沉淀”提升为正式持久化字段。

## 2026-03-27：阅读器快捷键先做高频主路径，并把提示直接放在界面里

状态：已生效

背景：

- 阅读器已经覆盖翻页、模式切换、搜索定位、批注和回项目等高频动作，但如果只能靠按钮点击，Windows 桌面上的连续阅读体验仍然偏慢。
- 纯粹增加快捷键而不做提示，会变成“隐藏能力”，对真实使用帮助有限。

决策：

- 这一阶段先只覆盖高频主路径快捷操作：聚焦定位、页切换、段落跳转、模式切换、批注保存、回到项目。
- 快捷键提示直接显示在阅读器界面里，不把它们藏进帮助弹层或额外设置页。
- 输入框与文本域内继续让位给正常输入；`Ctrl + Enter` 作为批注输入框里的显式提交动作保留。
- 阅读器壳层保持可聚焦，优先承接进入阅读后的键盘操作，不为 1H 引入更复杂的命令面板或全局快捷键系统。

影响：

- 键盘操作开始真正服务阅读主链路，而不是只停留在零散的翻页热键。
- 这让后续扩展图表优先流、命令面板或更细粒度快捷键时，有了清晰的最小可用基线。
- 通过把提示直接放在界面里，团队后续也更容易围绕真实可见的交互继续打磨，而不是遗忘这套能力。

## 2026-03-27：图表优先阅读流先复用已有图像提取和正文锚点，不新增重型布局

状态：已生效

背景：

- 对图示密集论文，很多用户的真实阅读顺序是先扫图建立全局判断，再回正文核对术语、实验设置和论证。
- 当前阅读器已经有图像提取、页预览、图像查看和段落定位能力，但还缺一条把这些能力串起来的显性流程。

决策：

- 阶段 1I 先在阅读器主区新增轻量的“图表优先阅读”卡片，不引入新的固定侧栏或双栏壳层。
- 图表优先流直接复用现有 `reader.figures`、页面预览和 `anchor_paragraph_id`，为每张图提供“先看图 / 回正文锚点”两步动作。
- 图示页的阅读提示直接整合进当前页概览和辅助文本提示，不单独做新的模式切换语义。
- 为了稳定验证这一阶段，E2E 种子论文补充真实图示页，而不是只依赖无图样本做空壳验证。

影响：

- 图示密集论文的主阅读路径更贴近真实习惯，而且实现范围仍然控制在现有阅读器能力之内。
- 这条路径为后续是否要做更完整的“图表总览 / 图表摘要 / 图表标注”能力留下了自然的演进点。
- 团队可以先观察这条轻量流程是否足够有用，再决定是否升级成更重的图表阅读工作台。

## 2026-03-27：阅读偏好先做本机级宽度与密度配置，不扩展成全局主题系统

状态：已生效

背景：

- 默认阅读体验已经通过排版、键盘和图表流得到一轮增强，但不同用户对正文宽度、密度和节奏的偏好仍会明显不同。
- 这轮目标是补最小可用的个性化能力，而不是把整个应用带进新的全局主题设计工程。

决策：

- 阅读偏好当前只覆盖阅读器局部的 `阅读宽度` 和 `阅读密度`，不扩展到全局色彩主题或全应用字号系统。
- 偏好按本机持久化保存，作用范围跨论文但不依赖具体项目或论文会话。
- 偏好控制入口直接放在阅读器当前页概览区，避免用户为了微调阅读体验跳到设置页中断当前阅读。
- 偏好样式只作用于辅助文本阅读容器，不改动其他页面或全局视觉基线。

影响：

- 用户可以在不离开当前阅读任务的前提下，把长文阅读调到更适合自己的宽度和密度。
- 这让阅读器保持“局部可调、全局克制”的策略，避免过早演化成高维护成本的主题系统。
- 如果后续观察到用户明确需要更多个性化项，再从这个局部偏好模型向更完整的阅读主题系统演进。

- 阶段目标与关键结论不再只散落在历史对话或临时文档里。

## 2026-03-27：阅读体验成为近期最高优先级体验改进项

状态：已生效

背景：

- 当前阅读器已经承载阅读、翻译、批注、证据提取和项目回流等关键动作。
- 用户明确反馈当前阅读体验仍然偏差。

决策：

- 近期体验优化优先围绕阅读器主链路展开。
- 在没有明显收益之前，不优先扩展与阅读主链路无关的新交互面。
- 评估改动时，优先看是否降低了阅读中断、切换成本和认知负担。

影响：

- 近期迭代应优先投入阅读会话恢复、焦点强化、导航和排版优化。

## 2026-03-18：项目工作台是产品主入口

状态：已生效

背景：

- 产品需要围绕具体研究问题组织工作，而不是把用户留在分散的模块入口中。

决策：

- 主入口保持为 `/projects`。

## 2026-03-18：坚持桌面优先主线

状态：已生效

背景：

- 当前主产品目标平台是 Windows 桌面，技术主线已经切到 Tauri。

决策：

- 产品保持 desktop-first，不回退为 web-first 心智。

影响：

- 界面与交互优先围绕桌面研究工作台设计。

## 2026-03-18：运行时数据路径区分开发态与安装态

状态：已生效

背景：

- 桌面应用需要同时支持仓库内开发运行和安装后独立运行。

决策：

- 仓库开发态运行时数据使用 `backend/data`。

影响：

## 2026-03-18：模型提供方配置保持可编辑

状态：已生效

背景：

- 研究工作流需要兼容不同模型和网关配置，且桌面端用户需要在 UI 中直接调整。

决策：

- 当前支持的 provider 模式包括 `openai`、`deepseek`、`openai_compatible`、`fallback`。
- Provider 设置尽量保持可在桌面设置页中编辑。
- OpenAI-compatible 网关遵循 OpenAI 风格请求语义。

影响：

- 接入新模型时优先复用现有 provider 抽象，而不是写散落的特例逻辑。
